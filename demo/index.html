From Brief to Outreach: A Workflow Wired in Public

One agent, four APIs, one job. A worked example of what an agent built on Parallel's surface actually does between the trigger event and the drafted reply.

Kevin Nguyen · May 2026 · ~7 min read
Methodology Sample outputs in this trace are constructed from Parallel's documented API surface. Call shapes, response schemas, and field names match the production API. Company names in the trace are fictional. The workflow is real; the run is illustrative — a faithful reconstruction of what an agent built on this stack does.
The Brief
"Wake me when a Series-B AI infrastructure company hires a new VP Engineering. I want an enriched company record, the new hire's background, and three grounded hooks for a same-day outreach — every fact traceable to a citation the SDR can paste back into the email."

The Pipeline · Four APIs, One Loop

Stage 01

Source the watchlist

SourceFindAll API ⏱ 11.2s · 50 candidates

The agent does not start with a CRM list. It starts with a brief. Stage 1 turns the brief into a verified list of entities that match — companies in the right stage, the right vertical, the right geography. No CSV upload, no scraped Crunchbase, no manual filter.

POST https://api.parallel.ai/v1/findall/runs

  "objective": "Series B AI-infrastructure startups, US-based,
                post-2022 founding, building infra not apps",

  "match_conditions": [
    "raised $30M-$100M Series B",
    "primary product is developer infrastructure",
    "US-headquartered",
    "founded 2022 or later"
  ],
  "match_limit": 50,
  "generator": "core"

Stage 02

Enrich each entity, with basis

ReasonTask API · Enrichment, processor=core ⏱ 1m 47s · 50 records

Stage 2 takes the brief from a list of names to a structured dataset. The schema is the contract. Every field carries its own citation and confidence — the agent can route low-confidence fields to a second pass without recomputing the rest.

Request — Group enrichment with explicit schema
POST https://api.parallel.ai/v1/task/groups
{
  "processor": "core",
  "inputs": [/* the 50 FindAll matches */],
  "output_schema": {
    "founded_year": "integer",
    "total_funding_usd": "integer",
    "series_b_lead_investor": "string",
    "headcount_estimate": "integer",
    "engineering_pct": "number",
    "primary_tech_stack": "string[]",
    "current_vp_engineering": "string | null"
  }
}

Stage 03

Watch for the trigger

WatchMonitor API · frequency=1h ⏱ Standing watch, 50 entities

Stage 3 is the most underrated primitive on the stack — and the one that turns the workflow from a one-shot job into a system. The agent does not poll Crunchbase. It registers an objective and gets a webhook when reality changes against it.

Request — one Monitor per company, batched
POST https://api.parallel.ai/v1/monitor/runs
{
  "objective": "new hire announcement for VP Engineering,
                Head of Engineering, or CTO at Cogent Labs",
  "frequency": "1h",
  "processor": "base",
  "webhook_url": "https://sdr-bench.kevin.energy/hooks/monitor",
  "context": {
    "entity_id": "cogent-labs",
    "watch_type": "vpe_hire"
  }
}
Response — event payload delivered to webhook (Wed 09:14 PT)
{
  "event_id": "ev_7Wgr...3qC",
  "monitor_id": "mon_4Pfn...2vK",
  "detected_at": "2026-05-13T09:14:21Z",
  "change_type": "role_filled",
  "output": {
    "summary": "Cogent Labs announced Maya Okafor as VP Engineering,
              previously Director of Infrastructure at Stripe.",
    "basis": [
      { "citation": "cogent.ai/blog/welcoming-maya", "confidence": 0.99 },
      { "citation": "linkedin.com/in/mayaokafor",        "confidence": 0.97 }

  }
}
Handoff → Webhook fires. Pipeline resumes. Stage 4 grounds the outreach in citations from the public surface before the SDR's coffee is cold.

Stage 04

Ground the outreach

PerceiveSearch API · mode=advanced ⏱ 4.8s · 3 grounded hooks

The agent does not need ten blue links about Maya Okafor. It needs three specific facts that an SDR can paste into an email without copying anything that isn't cited. Stage 4 is a precision query, not a research session.

Request — one objective, three keyword queries
POST https://api.parallel.ai/v1/search
{
  "objective": "Three specific, citable facts about Maya Okafor
                that would be useful for a same-day intro
                about agent infrastructure, retrieval, or DX.",
  "queries": [
    "Maya Okafor Stripe infrastructure director",
    "Maya Okafor public talks conference",
    "Maya Okafor opinions agent retrieval"
  ],
  "max_results": 3,
  "mode": "advanced"
}
Response — three excerpts, each cited
{
  "results": [
    {
      "excerpt": "... Okafor led the team that rebuilt Stripe's idempotency-key
                  infra in 2024, a project credited with cutting duplicate-charge
                  incidents by 78% ...",
      "citation": "stripe.com/blog/idempotency-2024",
      "confidence": 0.97
    },
    {
      "excerpt": "... 'Retrieval is the rate-limit on what agents can do safely'
                  — Maya Okafor, AI Engineer Summit, Oct 2025 ...",

What this workflow demonstrates
  • Five primitives, four APIs, one job. The essay argued web infrastructure for agents has five primitives. The workflow is what those primitives look like wired together — not as a demo, as a system.
  • Confidence routes the work. Stage 2 holds low-confidence enrichments for review without re-running the rest. Stage 3 only watches the entity-fields the agent is most sure about. Calibrated confidence is not a feature; it is the schedule.
  • The schema is the contract. Every stage either consumes a typed input or produces a typed output. Nothing in this pipeline is "look at the search results and figure it out." The handoff between stages is what makes the workflow composable rather than fragile.
  • Citations are the audit trail. Four facts in the final email, four citations in the artifact, four confidence scores attached. The SDR can paste the citations back in if a prospect asks where a claim came from. That is what production-grade web data feels like in the hand.
