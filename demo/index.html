Workflow Trace · Reconstructed Against a Real Trigger Event

From Brief to Outreach: A Workflow Wired in Public

One agent, four APIs, one job. A reconstruction of what an agent built on Parallel's surface would have produced on May 4, 2026 — the day Sierra's $950M Series E was announced.

Kevin Nguyen · May 2026 · ~7 min read
Methodology

Every company, person, fact, and citation in this trace is real and publicly documented. The trigger event is Sierra's $950M Series E announced May 4, 2026 at a $15.8B post-money valuation.

Sample API outputs are constructed from Parallel's documented surface — call shapes, response schemas, and field names match the production API. The workflow is real; the run is reconstructed using the actual public information available on the day of the trigger.

The Brief
"Wake me when a Series-B+ AI agent company announces a major funding round. Major rounds are 6-month buying windows for retrieval infrastructure. I want an enriched company record, three citable facts from the round coverage, and a same-day outreach the SDR can send — every claim traceable to a public source the prospect already trusts."

The Pipeline · Four APIs, One Loop

Stage 01

Source the watchlist

SourceFindAll API ⏱ 14.1s · 50 candidates

The agent does not start with a CRM list. It starts with a brief. Stage 1 turns the brief into a verified list of entities — companies in the right stage, the right vertical, the right geography. No CSV upload, no scraped Crunchbase, no manual filter.

Stage 02

Enrich each entity, with basis

ReasonTask API · Enrichment, processor=core ⏱ 2m 04s · 50 records

Stage 2 takes the brief from a list of names to a structured dataset. The schema is the contract. Every field carries its own citation and confidence — the agent can route low-confidence fields to a second pass without recomputing the rest.

Stage 03

Watch for the trigger

WatchMonitor API · frequency=1h ⏱ Standing watch, 50 entities

Stage 3 is the most underrated primitive on the stack — and the one that turns the workflow from a one-shot job into a system. The agent does not poll Crunchbase. It registers an objective and gets a webhook when reality changes against it.

Request — one Monitor per company, batched
POST https://api.parallel.ai/v1/monitor/runs
{
  "objective": "new funding-round announcement, equity round of $50M+,
                for Sierra (sierra.ai)",
  "frequency": "1h",
  "processor": "base",
  "webhook_url": "https://sdr-bench.kevin.energy/hooks/monitor",
  "context": {
    "entity_id": "sierra",
    "watch_type": "major_funding_round",
    "min_round_size_usd": 50000000
  }

Stage 04

Ground the outreach

PerceiveSearch API · mode=advanced ⏱ 5.3s · 3 grounded hooks

The agent does not need ten blue links about Sierra. It needs three specific facts an SDR can paste into the email without copying anything that isn't cited. Stage 4 is a precision query, not a research session.

What this workflow demonstrates
  • Five primitives, four APIs, one job. The essay argued web infrastructure for agents has five primitives. The workflow is what those primitives look like wired together — not as a demo, as a system.
  • Confidence routes the work. Stage 2 holds low-confidence enrichments for review without re-running the rest. Stage 3 only watches the entity-fields the agent is most sure about. Calibrated confidence is not a feature; it is the schedule.
  • The schema is the contract. Every stage either consumes a typed input or produces a typed output. Nothing in this pipeline is "look at the search results and figure it out." The handoff between stages is what makes the workflow composable rather than fragile.
  • Citations are the audit trail. Three facts in the final email, three citations in the artifact, three confidence scores attached. The SDR can paste the citations back in if a prospect asks where a claim came from. Every link in this trace resolves to a real public page that supports the claim. That is what production-grade web data feels like in the hand.
